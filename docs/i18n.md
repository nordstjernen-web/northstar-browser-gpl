# Internationalisation in Nordstjernen

Nordstjernen renders web content in any script that has a font
installed on the host — Latin, Cyrillic, Greek, Arabic, Hebrew,
Devanagari, Thai, the CJK block (Simplified Chinese, Traditional
Chinese, Japanese, Korean), Vietnamese, and the rest. There is no
list of "supported languages" maintained in the source; the
browser leans on the platform text stack and reads whatever the
fonts can draw.

The UI is written in English and **translated to the operating-system
language at startup** through a small in-tree catalogue lookup — see
[UI translation](#ui-translation) below. The rest of this document
covers rendering of *page content*, which is a different concern.

## UI translation

`src/i18n.c` is the whole translation library: ~120 lines, no gettext,
no `.po` tooling.

- **Catalogues** live in `data/i18n/<lang>.lang` (installed to
  `<datadir>/nordstjernen/i18n/`). The format is one
  `English source=translation` pair per line, UTF-8, `#` comments.
  Forty catalogues ship today — the 40 most spoken languages
  (en, zh, hi, es, fr, ar, bn, pt, ru, ur, id, de, ja, mr, te, tr, ta,
  vi, ko, it, th, fa, pl, uk, nl, sw, tl, pa, gu, kn, ml, my, am, ha,
  yo, jv, su, ro, el, hu) — each translating the word **Nordstjernen**
  (the North Star: de `Polarstern`, ru `Полярная звезда`, ja `北極星`,
  hi `ध्रुव तारा`, …). Thirteen of them (de, fr, es, pt, it, nl, ru,
  uk, pl, tr, ja, zh, ko) additionally cover the core UI vocabulary —
  toolbar buttons, menus, tabs, bookmarks, settings, status messages.
- **Language pick**: `ns_i18n_init` walks `g_get_language_names()`
  (the OS locale: `LANGUAGE` / `LC_ALL` / `LANG`, or the Windows user
  language through GLib) and loads the first catalogue that exists;
  `NS_I18N_DIR` overrides the search directory, and source-tree runs
  find `data/i18n/` relative to the executable.
- **Lookup**: `ns_i18n("...")` returns the translation or the English
  input unchanged — a missing catalogue or missing key silently falls
  back, so English remains the source of truth. All user-visible GTK
  shell strings (menus, toolbar tooltips, context menus, settings,
  bookmarks, task manager, status messages, dialogs) route through it.
  Strings that are `printf` formats keep a literal format and translate
  the words via `%s` arguments (`-Wformat=2` stays happy).
- Under a German locale the window titles itself
  `<page> — Polarstern`; under an untranslated or `C` locale everything
  stays English.

To add a language: drop a `data/i18n/<code>.lang` file (the `<code>`
must match a `g_get_language_names()` entry, e.g. `nb` or `pt_BR`) and
add it to the `install_data` list in `meson.build`. To translate more
of the UI, add `English source=translation` lines — the English string
used in the source code is the key.

## Text stack

Nordstjernen draws all page text through Pango and Cairo, which
GTK 4 brings in unconditionally. Underneath:

| Layer        | Library                       | Role                                               |
| ------------ | ----------------------------- | -------------------------------------------------- |
| Shaping      | **HarfBuzz** (via Pango)      | OpenType glyph selection, ligatures, marks         |
| Bidi         | **FriBidi** (via Pango)       | Unicode Bidirectional Algorithm (UAX #9)           |
| Line break   | Pango                         | UAX #14 line break, script itemisation             |
| Font lookup  | **Fontconfig** + **FreeType** | System font discovery, fallback chains             |
| Charset      | declared label + **uchardet** | Honours a BOM, the HTTP `Content-Type` charset, and an in-page `<meta charset>`; falls back to uchardet detection, then `g_convert` to UTF-8 |
| Text input   | **GtkIMContext** (multicontext) | Routes composed input (Pinyin/Wubi/kana/Hangul) from the platform input method into form fields |

HarfBuzz is **not** an explicit dependency in `meson.build` — it
arrives transitively as part of Pango (`pangoft2` /
`pango-cairo`), which GTK 4 pulls in. There is nothing to enable
or configure: Arabic and Indic shaping work out of the box on any
system that has GTK 4 installed.

## What the browser actually does for i18n

1. **UTF-8 throughout.** Response bodies are decoded to UTF-8 by
   `ns_html_decode_body_full` in `src/html.c` before they reach the
   parser, following the HTML spec's priority order: a leading BOM,
   then the HTTP `Content-Type` charset, then an in-page
   `<meta charset>` (the whole body is scanned, so legacy Chinese
   GBK/GB18030 and Big5 pages that open with tens of kilobytes of
   ASCII script before any high bytes still decode correctly), and
   only then uchardet statistical detection as a fallback. A small
   normalisation table maps the labels that differ from their iconv
   names, following the WHATWG encoding standard:

   - **Chinese:** `gb2312` is widened to `GBK`; `big5` and
     `big5-hkscs` convert directly.
   - **Cyrillic** (legacy RuNet): `x-cp1251` → `WINDOWS-1251`,
     `koi8_r` → `KOI8-R`, `x-mac-cyrillic` / `x-mac-ukrainian` →
     `MAC-CYRILLIC` (the standard `windows-1251`, `cp1251`, `koi8-r`,
     `koi8-u`, `ibm866`, `iso-8859-5` labels already convert).
   - **Japanese:** every `shift_jis` alias (`shift_jis`, `sjis`,
     `x-sjis`, `ms_kanji`, `csshiftjis`, `windows-31j`) maps to
     **`CP932`**, the Windows-31J superset — so the Microsoft
     extensions real Japanese pages depend on decode correctly: the
     `shift_jis` byte `0x8160` becomes the fullwidth tilde U+FF5E (not
     the bare wave dash U+301C), and the NEC/IBM rows (`①` U+2460,
     `㈱` U+3231, …) resolve instead of failing. `x-euc-jp` → `EUC-JP`;
     `euc-jp` and `iso-2022-jp` convert directly.

   The DOM, layout tree, and Pango layouts all hold UTF-8.

   Forms submit in the **document's encoding** (or the form's
   `accept-charset`): a search box sends GBK bytes on a GBK page,
   cp1251 bytes on a windows-1251 page, and CP932 bytes on a
   `shift_jis` page (`検索` → `%8C%9F%8D%F5`), percent-escaped — which
   legacy Chinese, Russian, and Japanese sites expect
   (`ns_form_set_submission_charset` in `src/net.c`).

   **Text entry in CJK and other composed scripts** flows through the
   platform input method: a `GtkIMContext` on the GTK
   page view (`src/gtk/procview.c`). Fcitx/
   IBus with Pinyin, Wubi, Mozc, Hangul, … composes off-screen and the
   committed string is inserted into the focused field over the existing
   key IPC. Printable keystrokes are routed to the IM rather than
   inserted directly so a composing session never double-types.
2. **Script itemisation and shaping.** Each `PangoLayout` is fed
   the raw UTF-8 string; Pango runs script detection, then hands
   each run to HarfBuzz, which picks the right OpenType shaper
   (Arabic, Indic, Hebrew, default, etc.).
3. **HTML `lang` attribute.** Walked up from the text node to the
   nearest ancestor that carries a `lang=` attribute, then applied
   to the layout as a `pango_attr_language_new(...)` that spans
   the whole run. The main reason this matters in practice: CJK
   fonts often contain glyphs for both Simplified and Traditional
   Chinese (or Japanese) at the same code point, and Pango picks
   the right regional variant based on the language tag.
4. **HTML `dir` attribute.** Same upward walk. `dir="rtl"` and
   `dir="ltr"` flip the layout's base direction explicitly via
   `pango_context_set_base_dir` and disable Pango's
   auto-detection. When `dir` is absent, Pango's default
   auto-direction kicks in and chooses the paragraph direction
   from the first strong directional character — which is the
   right thing for most pages.
5. **Bidi.** Done by FriBidi inside Pango. Nordstjernen has no
   bidi code of its own.
6. **Font fallback.** Done by Fontconfig. If the requested font
   does not contain a glyph for a given codepoint, Pango asks
   Fontconfig for a substitute and re-shapes that run. This is
   how a Latin-default page with one Chinese character still
   renders that character without us doing anything.
7. **`@font-face` web fonts.** Loaded over HTTP by `src/font.c`
   and installed into the Fontconfig app-font set; from that
   point Pango sees them like any system font.

## Internationalised domain names (IDN)

URLs with non-ASCII hostnames work end-to-end. Typing
`https://википедия.рф/` in the URL bar (or following such a link)
goes through lexbor's WHATWG URL parser, which applies UTS #46
+ Punycode (RFC 3492) and hands libcurl the ASCII form
(`https://xn--b1aeclack5b4j.xn--p1ai/`). The wire and DNS lookup
are always ASCII; the IDN code lives in lexbor's
`unicode/idna.c` and `punycode/`.

The URL bar shows the Unicode form when it is **unambiguous**:

- pure-ASCII labels: shown as-is
- labels whose non-ASCII codepoints are all in a single script
  from {Cyrillic, Greek, Han, Hiragana, Katakana, Hangul, Arabic,
  Hebrew, Devanagari, Thai, …}: shown in Unicode

When a label *mixes* Latin, Cyrillic, and Greek codepoints
(`страße.de`, `pаypal.com` with a Cyrillic `а`), the URL bar
falls back to the ASCII Punycode form so the user sees what was
actually resolved.

**Limitation:** the check does not catch all-Cyrillic homographs
of Latin labels (`аррӏе.com` built from Cyrillic а р р ӏ е),
because detecting those needs Unicode confusables tables (the
kind ICU ships). Adding ICU just for this is a bigger trade-off
than the project wants to make; for now this class of spoof is
documented rather than blocked. Copy the URL out of the bar to
see whether the resolved host is really ASCII Punycode.

The helpers are `ns_url_to_ascii` (always-Punycode, used before
handing the URL to libcurl) and `ns_url_to_display` (Unicode if
safe, else Punycode, used for the URL bar) in `src/net.c`.

## What is *not* done

- **No gettext.** UI translation goes through the in-tree
  `src/i18n.c` catalogue lookup (see above); built-in pages
  (`about:start`, error pages) and the catalogues' untranslated
  strings remain English.
- **No CSS `writing-mode: vertical-rl`** or other vertical
  text. Horizontal text only.
- **No `text-orientation`, `text-combine-upright`,** or other
  CJK-specific typographic CSS.
- **No `<bdo>` element handling** beyond Pango's default
  treatment. Explicit `dir` overrides via the `dir` attribute on
  any element are honoured.
- **No language-aware hyphenation.**
- **No language-aware case mapping** beyond what GLib's
  `g_utf8_strup` / `g_utf8_strdown` provides (which is locale
  independent and gets the Turkish dotted-I case wrong, like most
  programs).

## Required fonts

The browser draws whatever the system can find. On a typical
desktop install, the fonts most commonly required to display the
modern public web are:

| Script(s)                  | Font package (Debian / Ubuntu) |
| -------------------------- | ------------------------------ |
| Latin / Cyrillic / Greek   | `fonts-dejavu` (usually preinstalled) |
| Arabic                     | `fonts-noto-core` or `fonts-noto`     |
| Hebrew                     | `fonts-noto-core` or `fonts-noto`     |
| CJK (Hans / Hant / JP / KR)| `fonts-noto-cjk`                       |
| Devanagari / Tamil / Thai  | `fonts-noto-core`                      |
| Emoji                      | `fonts-noto-color-emoji`               |

On Fedora / RHEL: `google-noto-sans-fonts`,
`google-noto-sans-cjk-fonts`, `google-noto-color-emoji-fonts`.
On openSUSE: `noto-sans-fonts`, `noto-sans-cjk-fonts`,
`noto-coloremoji-fonts`. macOS and Windows ship comprehensive
font sets out of the box and need no extra packages for the
languages above.

If a glyph shows up as `.notdef` (the "tofu" box), the browser
has the right data but the host is missing the font — install
the relevant Noto package and reload.

## Smoke test

A quick way to confirm the end-to-end path works on a given
host:

```html
<!doctype html>
<html><body>
<p lang="ja">日本語のテキスト</p>
<p lang="zh-Hans">简体中文</p>
<p lang="zh-Hant">繁體中文</p>
<p lang="ar" dir="rtl">مرحبا بالعالم</p>
<p lang="he" dir="rtl">שלום עולם</p>
<p lang="ru">Привет, мир</p>
<p lang="ko">안녕하세요</p>
</body></html>
```

Save as `i18n.html` and either open it in the browser or render
with the headless driver (the interactive GUI gates `file://` off
by default, but `--headless` enables file and bare-path inputs):

```sh
nordstjernen --headless --dump=png:out.png i18n.html
```

The Arabic and Hebrew lines should be right-aligned, every line
should be made of real glyphs (no tofu), and the Simplified vs.
Traditional Chinese paragraphs should differ in places where the
two regional standards use different glyph shapes for the same
codepoint.
