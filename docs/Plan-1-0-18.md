Nordstjernen 1.0.18 — development-cycle plan
============================================

Snapshot: **1.0.18-dev**, 2026-06-30. A prioritized focus list for the
1.0.18 cycle, derived from a fresh pass over the source tree, the WPT
scoreboards, and the carry-over tail from the (now-deleted) 1.0.17 plan.
Every item is anchored to concrete `file:line` evidence so it can be
re-checked before it is picked up.

The cycle opens off the back of a layout-engine correctness pass (the
1.0.17→18 boundary fixed two CSS sizing bugs: grid `auto` track sizing,
commit `f696c7e`, and per-line flex-grow in wrapping flex rows, commit
`58a35e8`). That work surfaced the theme that anchors this cycle: **the
layout engine is where real-world rendering breaks, and each fix is small,
verifiable, and high-impact.** The project ships **no automated test suite
by policy**, so every change remains a manual-verify event — prefer fixes
that are reproducible head­lessly with `--headless --dump=layout|text`
against a minimal repro.

Status legend: severity (high/med/low) · effort (small/med/large) · ROI.


## What landed at the 1.0.17→1.0.18 boundary

- **Grid `auto` track sizing** (`f696c7e`). An `auto` track was based on
  its max-content, so a long paragraph in `grid-template-columns: auto
  330px` blew past the container (over-wide Wikipedia donation banner) or
  collapsed to one word per line. Auto tracks now base on min-content and
  grow toward max-content capped by free space, then stretch.
  (`resolve_track_sizes_full`, `layout.c`.)
- **Per-line flex-grow in wrapping flex rows** (`58a35e8`). A `flex-wrap`
  row laid every item at its flex-basis and spent leftover space only on
  justify spacing — flex-grow was never applied, so `flex: 1 0 0`
  collapsed to width 0 (small "Wikipedia can't be sold" banner).
  `layout_flex_row_wrap` now distributes each line's free space to its
  growable items.

These two are the proof of the cycle thesis. The ranked list below is
weighted toward the same kind of work.


## Headline themes

1. **Layout-engine correctness is the highest-leverage, lowest-risk
   area.** Two one-function fixes this boundary corrected real top-site
   rendering. The follow-ons — container-query evaluation depth, flex
   baseline alignment, subtree-local relayout — are concentrated in
   `layout.c` (10,696 lines) and `css.c`, are reproducible headlessly, and
   pay off on every page, not just WPT.

2. **URL/IDNA is still the single biggest *honest* WPT lever, and it
   grew.** `IdnaTestV2` is now 2671 subtests with ~800 failing — the
   largest single bucket in the slice (`docs/wpt-scores.md:153`). The
   in-tree lexbor `unicode/idna.c` is heavily locally modified, so the
   failures are partly self-inflicted. Carried over from 1.0.17 rank 6,
   unstarted.

3. **`src/js.c` is now 40,761 lines and still growing** (+388 since the
   1.0.17 snapshot). CLAUDE.md elevates "readable and auditable by a
   single human" to a *design constraint*; the recurring security findings
   land in this file. The scoped workers→`js_worker.c` extraction
   (1.0.17 rank 5) never happened — do it this cycle before the file
   grows further.

4. **Finish what the API surface advertises but doesn't deliver.** Module
   workers still throw (`js.c:16614`), `video.currentTime` is a no-op
   setter (`js.c:30640`), `getByteFrequencyData` throws instead of
   returning zeros (`js.c:8601`), `createConicGradient` is absent. Each is
   contained; each is a latent "lies about support" bug.

5. **Stop the doc rot.** `docs/CSS-compatibility.md` is snapshot
   **1.0.10** (2026-06-18) — eight releases stale (`:15`). An out-of-date
   capability matrix actively misdirects. Cheap to fix, high signal.

## Ranked initiatives

### 1 — Audit `@container` query evaluation depth  ·  layout · high · med · high

Container queries are *parsed* — `container-type`/`container-name`
(`css.c:282-283`, decl parse `css.c:7916-7928`) and the `cqw/cqh/cqi/cqb/
cqmin/cqmax` units (`css.c:2572-2597`) all exist — and the small
"can't be sold" banner this boundary used `container-type: inline-size`
and rendered correctly once flex-grow was fixed. That makes container
queries a *shipped* feature whose evaluation correctness has never been
audited.

- **Why:** container queries are now standard in modern responsive CSS
  (every WMF banner, most component libraries). A subtly-wrong query
  context produces wrong layout silently — exactly the failure class the
  no-test-suite policy hides.
- **First steps:** build headless repros that (a) resolve a size query
  against the *nearest* `container-type` ancestor (not the viewport),
  (b) exercise `@container name (min-width: …)` named lookups,
  (c) verify `cqw`/`cqi`/`cqmin` resolve to the query container's box, not
  the ICB. Diff observed widths against Chrome on the same minimal pages.
  Fix the evaluator where it diverges; keep each repro in
  `data/render-tests/`.

### 2 — Fix the in-tree lexbor IDNA/UTS-46 path  ·  compliance · large · high

Carried over from 1.0.17 rank 6, unstarted, and the bucket grew.

- **Evidence:** `docs/wpt-scores.md:114` `url` = 1027 available gain, top
  of the ROI table; `:153` notes `IdnaTestV2` is now 2671 subtests with
  ~800 failing — the largest single bucket. The in-tree lexbor
  `unicode/idna.c` is heavily locally modified, so a share of the failures
  is our own regression surface.
- **First steps:** `scripts/wpt-score.sh --wpt-root=~/wpt url` to isolate
  failing `IdnaTestV2`/`toascii` subtests; diff `unicode/idna.c` against
  upstream lexbor 3.1.0 to find local-modification regressions; fix the
  UTS-46 mapping/validity path; rerun just `url` and commit the
  regenerated scoreboard with the engine change. **Budget the full WPT
  re-run as the real cost.**

### 3 — Extract the workers subsystem out of `src/js.c`  ·  maintainability · med · med

The biggest auditability violation, and it grew this cycle.

- **Evidence:** `wc -l src/js.c` = 40,761 (was 40,373 at the 1.0.17
  snapshot); the next-largest source is `css.c` at 15,760. `js.c` owns
  timers, workers, XHR/fetch, custom elements, all DOM bindings, WebCrypto
  marshalling, iframe scoping and the JS pump. The project already splits
  `js_intl.c` / `js_canvas.c` / `js_perf.c` via `subdir()`.
- **First steps:** carve `ns_worker_*` + the worker realm bridge into
  `src/js_worker.c` with a small internal header for the shared statics it
  touches; wire via `subdir()` exactly as `js_canvas.c` is; confirm clean
  compile + headless smoke. No behaviour change. Doing this *before* rank 4
  (module workers) means the new code lands in the new file. Defer
  `js_fetch.c` / `js_ce.c` to later cycles.

### 4 — Module workers  ·  feature · med · med

Carried over from 1.0.17 rank 9, unstarted; pairs naturally after rank 3.

- **Evidence:** `src/js.c:16614` throws
  `TypeError: Worker: module workers are not supported` for any non-classic
  type. Dedicated classic workers (with `importScripts`/`fetch`/
  `crypto.subtle`) already work, and the engine already runs module scripts
  on the main realm — both halves exist; the increment is joining them.
- **Why:** `new Worker(url, {type:'module'})` is the Vite/bundleless-ESM
  default, so the classic-only limitation silently breaks a growing share
  of real sites. SharedWorker stays unsupported.
- **First steps:** accept `type:'module'` in the Worker constructor and
  route the entry script through the existing module-evaluation path in the
  worker realm; verify with a `new Worker(new URL('./w.js',
  import.meta.url), {type:'module'})` that imports a sibling ESM module.

### 5 — URL long tail  ·  compliance · med · high

The non-IDNA half of the `url` bucket, mostly small lexbor fixes.

- **Evidence:** `docs/wpt-scores.md:153` lists lexbor opaque-path erasure
  (`pathname=''` on non-special URLs), non-special host removal,
  host-setter port overflow, and trailing-space serialization on opaque
  paths under `url-setters`/`url-constructor` (~1000 with IDNA, a few
  hundred without).
- **First steps:** run `scripts/wpt-score.sh url`, pick the
  `url-setters`/`url-constructor` subtests that are *not* `IdnaTestV2`,
  and fix them in the lexbor URL module (these are independent of rank 2
  and can land first as quick wins).

### 6 — `scroll`/`scrollend` event infrastructure  ·  feature/compliance · med · med

A new lever that surfaced in the 2026-06-29 WPT run.

- **Evidence:** `docs/wpt-scores.md:151` — ~26 `dom/events/scrolling`
  files need real scroll + `scrollend` (programmatic scroll, scroll-snap,
  `scrollIntoView`, arrow-key scroll); each waits on a scroll/scrollend
  event the engine never generates, so the pages time out.
- **Why:** beyond WPT, `scrollend` and reliable programmatic
  `scrollIntoView` are used by real single-page apps for focus management
  and infinite scroll. The engine has scroll position; it just doesn't emit
  the DOM events.
- **First steps:** emit `scroll` on programmatic and user scroll, and
  `scrollend` when scrolling settles; back `element.scrollIntoView()` with
  a real position change + event; verify against the `scrolling/scrollend-*`
  fixtures.

### 7 — Flex baseline alignment  ·  layout · med · med

The one well-known flex gap, deferred in 1.0.17 as "not a quick win."

- **Evidence:** the 1.0.17 plan flagged `align-items: baseline` as needing
  each item's first line-box ascent — a post-layout used value bolted onto
  the deliberately single-pass flex implementation (`layout.c`, the
  `layout_flex_row` family around the assigned-main loop). With the
  flex-wrap grow path now corrected (this boundary), the flex code is fresh
  in scope.
- **First steps:** after the per-item main-size layout pass, read each
  item's first baseline (ascent of its first line box) and align the line
  to the max ascent for `align-items/align-self: baseline`. Reproduce
  headlessly with a row of mixed font-size items; this is a self-contained
  second pass, not a rewrite.

### 8 — Small high-confidence correctness bundle  ·  correctness · med · high

Self-contained fixes, each closing a "surface advertises support it
doesn't deliver" gap. All carried over from 1.0.17 and still open.

- **`video.currentTime` seek:** `js.c:30640` binds `currentTime` to
  `ns_element_noop_set` (a no-op). Wire a `seek <token> <t>` command from a
  `currentTime` assignment to `av_seek_frame` + the audio helper.
  (Crosses the renderer↔audio boundary and is not headless-verifiable
  here — pair it with a manual GUI smoke on an `.mpg`/`.webm`.)
- **`createConicGradient`:** absent from `src/` entirely; `conic-gradient()`
  already exists in the CSS path. Bind it in `js_canvas.c` — closes the
  last well-known Canvas 2D gap.
- **`getByteFrequencyData`→zeros:** `js.c:8601` binds it to
  `ns_audio_analysis_throw`. Returning zeros (instead of throwing) lets
  loose `try`/`catch` audio-visualizers degrade gracefully instead of
  breaking — an honesty fix, not a Web Audio commitment.
- **De-truncate the depth-512 DOM walkers:** `NS_DOM_MAX_DEPTH = 512`
  (`dom.c:1011`) truncates script-reachable walkers — `clone` (`:1016`),
  the tag/id/class index builders (`:1156`, …) — so deep documents get
  wrong `textContent`/serialization/clone and elements invisible to
  `getElementById`. Convert the half-dozen script-reachable walkers to
  iterative + explicit-stack (as `ns_node_free` already is).

### 9 — Doc hygiene  ·  docs · small · high

- **Evidence:** `docs/CSS-compatibility.md:15` is snapshot **1.0.10**
  (2026-06-18), eight releases stale; the WPT Top-10
  (`docs/wpt-scores.md:130`) is dated 2026-06-27 and several of its rows
  already landed.
- **First steps:** re-check every ❌/🟡 row in `CSS-compatibility.md`
  against current source (verify container-query, grid `auto`, and
  flex-wrap rows reflect this boundary's fixes; downgrade flex baseline to
  🟡 pending rank 7) and bump the snapshot; re-cluster the WPT Top-10 now
  that ranks 1–6 of the old list have landed.
